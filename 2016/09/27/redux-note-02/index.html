从redux开始学习react:

React的这套机制,之所以那么有特色(也可以说不那么容易被理解),就是因为他所提出来的flux数据流的概念.

大家都知道React 只是一层简单的view层,既然只是一层view层,那什么才是最好的管理这个view层的方式呢?react中的数据交互,ajax应该放到哪里去实现呢?

目前为止看Dan Abramov在flux的理论基础上改进出的Redux是发挥React组件化威力的最佳搭档,既然react是简单的且可以被其他view框架替代的,那么不如先把redux单向数据流的概念搞清楚,再来看react一切会清晰很多.

Redux?

Redux is a predictable state container for JavaScript apps.

redux是 JavaScript App 中一个用于可预测state的容器(管理器),为什么是可预测的呢?怎么理解?
带着问题去找答案.

Redux 除了和 React 一起用外，还支持其它界面库。
它体小精悍（只有2kB）且没有任何依赖。

他有三大原则:

  • 单一数据源
  • State 是只读的
  • 使用纯函数来执行修改

单一数据源

  • Redux中所有的 state 都以一个对象树的形式储存在一个单一的 store 中。
  • 通过 store.getState() 可以获取这个对象

State 是只读的

  • 惟一改变 state 的办法是触发 action，action是一个描述发生什么的对象。
  • action 必须要有type属性(默认大写type属性),其他属性都可以自定义.

使用纯函数来执行修改

  • reducers 根据触发的 action 来决定如何改变 state 树。

Reducer 只是一些纯函数，它接收先前的 state 和 action，并返回新的 state。在一个大的应用中,你可以把它拆成多个小的 reducers，分别独立地操作 state tree 的不同部分.
